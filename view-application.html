1. Introduction

This Software Architecture Document provides an architectural overview of the proposed "Doctor Little" Workforce Optimization Platform.

2. General Context

2.1. Objectives

The "Doctor Little" application aims to provide cutting-edge healthcare technology solutions that enhance patient care, improve medical processes, and offer a more efficient and personalized healthcare experience.

2.2. Existing

The application is designed to build upon and transform an existing workforce management system, tailoring it to the specialized needs of healthcare providers and patients.

2.3. Positioning in the IS

The "Doctor Little" application sits within the larger context of healthcare information systems, focusing on healthcare provider and patient interactions and optimizing scheduling and patient care.

3. Legals

4. Requirements

4.1. Customer Needs & Requests

The application aims to address the following customer needs and requests:

  • Improvement of heavily fragmented and ineffective ways of working.

  • Integration of management systems for recruitment, onboarding, shift allocation, performance management, and more.

  • Rotation management and staff exchange between healthcare institutions.

  • Enhanced communication channels for better collaboration.

  • Dashboard with forecast capabilities and business intelligence.

  • Integration with hospital management systems and accounting (Payroll).

  • Multi-language support: Spanish, French & English.

4.2. Quantitative Customer Data

The application targets small and medium-sized healthcare institutions with an average capacity of 50-60 beds, employing an average of 100-120 staff members.

4.3. Personas

  • Healthcare Administrators

  • Clinical Staff

  • Non-Clinical Staff

5. Functional Requirements

The application’s functional requirements are as follows:

  • Workforce Management (WFM): Includes time and attendance, absence management, scheduling, payroll, forecasting, optimization, and AI-enabled recommendations.

  • Employee Experience (EX): Involves rewards & recognition, communication, and wellness features.

6. Target Architecture

6.1. General Application Architecture

The application follows a client-server architecture, with the frontend providing user interfaces and the backend managing data storage, processing, and business logic.

6.2. Detailed Application Architecture

The detailed architecture comprises three main components:

  • Frontend: Developed using React and React Native for web and mobile applications.

  • Backend: Utilizes Python and Fastapi to implement APIs for user interactions.

  • Database: Utilizes a relational database (e.g., PostgreSQL) to store user profiles and relevant data.

6.3. Principles that Dictated the Choices

The architectural choices prioritize modularity, scalability, and security. Microservices architecture is employed for the backend to ensure flexibility and easier maintenance.
